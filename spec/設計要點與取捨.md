# 設計要點與取捨

> 記錄 w-web-task 開發中的設計決策、取捨與「已知不修」事項, 供後續測試與維護遵循 (對應全域 §3.1 真痛/假痛三條件、§2.3 簡潔優先). 本版已納入設計對抗性審查 (review-design) 結論.

## 決策

### D1. 任務採扁平原子模型, 不做巢狀 multi-task
- **決策**: DB 的 task = 一收一回一終態的原子單元. 「多任務」以同頻道多個扁平 task 表達, 不在單一 task 內存多個 subtask 各自 state.
- **理由**: agent (Claude) 注意力有限, 同時維護多 sub-state 易出錯; 扁平模型涵蓋所有多任務場景 (詳 `設計_agent協定.md` §5) 且 DB 極簡.
- **影響**: 不提供 subtask 巢狀 API; 請求端要拆成多個 task 或一個 task + markdown 步驟.

### D2. 對外 agent 面用 HTTP REST, 對內前端用 kpFunExt
- **決策**: agent 輪詢走 HTTP REST (`apis`); 前端即時操作走 kpFunExt (WebSocket RPC) + useDbOrm 直通. 兩者共用同一 procCore 業務函式.
- **理由**: 輪詢無需持久連線, HTTP 對任意 CLI runtime (curl/fetch) 最易呼叫; 前端純 CRUD 表 (channels/channelMembers) 走 ORM change 即時同步, kpFunExt 較佳.
- **同步策略 (重要)**: tableNamesSync 須與 tableNamesExec 對齊 (僅 channels/channelMembers). messages/tasks **不放 sync** — 它們不在 tableNamesExec (寫入須走 kpFunExt 業務函式保 kmx/守衛/身分權威), 若放進 sync, 同步引擎收到其 tag 變更時找不到對應 select binding → client refreshTable 報 error + 洩漏未解 promise. 故 messages/tasks 前端一律以 `$fapi.getMessages/getTasks` **按需拉取** (選頻道/發送後重抓, 存本地渲染), 非 ORM change 推送.
- **開窗輪詢 (2026-07-06 業主要求補強)**: 純按需拉取會使「聊天視窗開啟期間, agent 的回應不會自動出現, 須切頻道才看得到」. 補強: ChatView 掛載期間每 5 秒靜默輪詢 (`pollRefresh`: 重拉 getMessages/getTasks, 資料有變才更新本地與重繪; 原本捲動貼底才自動捲到底, 不打斷使用者往上閱讀; 背景分頁不輪詢). 仍不走 ORM sync, 上述 select binding 限制不變.
- **影響**: 同一業務有兩個傳輸入口, 認證方式不同 (HTTP→getAndVerifyAppUser; kpfun→verifyConn 連線層).

### D3. 認證委外 SSO, 本套件不存 users/tokens
- **決策**: 沿用 w-web-api/perm 模式, token 由 SSO 核發, 本端 `getUserByToken` 驗證. 開發 `token=sys` 捷徑 (NODE_ENV!==production).
- **理由**: 需求明示「需接入外部驗證例如 sso」; 不重造認證輪子.
- **影響**: 無本地登入流程; 部署需設 `ssoBaseUrl` / `ssoAppToken`; 測試 mock getUserByToken.
- **agent 身分合約 (2026-07-07 業主確認)**: agent 為機器角色, 建議以 SSO **app token** (tokens 表 isApp='y', userId 欄=agent 識別碼) 認證 — SSO 解析回虛擬使用者 (id=token.userId, name/email 為非空占位值, isAdmin='y'), 大量 agent 無須逐一建真人帳號/email. 一般使用者 token 亦可作 agent 用. 兩者皆須通過 `checkUser` (id/email/name/isAdmin 四欄非空) — `getAndVerifyAppUser` 與 client 路徑對稱 (2026-07-07 補齊, 原僅 iseobj 檢查); app token 之占位值天然滿足, 不受影響. 已以真 SSO 鏈路端到端驗證 (SSO 發 app token → 解析 → checkUser → 輪詢/認領/回報全流程 + 無效 token fail-closed).

### D12. 已知結構性效能上限: ORM 全表掃描 × 輪詢熱路徑 × 無封存 (2026-07-07 文件化)
- **現況**: 底層 w-orm-lmdb 之 select() 為「getRange 全表讀入記憶體 → mingo 過濾」, 無索引查詢 — 即使 select({id}) 單筆查找亦掃全表. getRecentMessages/getRecentTasks 為 agent 每輪輪詢必經, 其成本正比於「全系統 messages/tasks 總筆數」而非該頻道筆數; messages/tasks 無封存機制, 筆數單調成長.
- **量級**: 總成本 ≈ agent 數 × 輪詢頻率 × 全庫筆數. 千筆級掃描為毫秒級無感; 十萬筆級單次掃描可達百毫秒級, 多 agent 每 30s 輪詢時開始有感, 且每輪全表 deserialize 有 GC 壓力.
- **取捨**: 單機 + 中小量級 (數萬筆內) 為本套件合約內場景, 現況可用; 屬「擴展上限」非現行 bug.
- **中長期改善方向**: ①w-orm-lmdb 支援以 channelId 為前綴之複合 key 範圍查詢 (成本改正比於該頻道筆數); ②封存機制 (done 任務/舊訊息移冷表, 熱表僅留活躍資料).
- **方向②已實作 (2026-07-07, 業主拍板)**:
  - 規則: 終態 (done/error) 且 timeDone 逾 `archiveAfterDays` (settings.json, 預設 30, <=0 停用) 之任務、timeCreate 逾齡之訊息 → 由 `procCore.archiveSweep` 搬至 `tasksArchive`/`messagesArchive` 冷表; 守衛「任務相關訊息 (kind=task/taskReply) 不早於其任務離開熱表」.
  - 觸發: WWebTask 初始化註冊 timer (啟動後 30s 首掃 + 每小時一次, locking flag 防重入); timer 不放 procCore → unit 測試可獨立建構退出 (SSO ADR-013 之教訓). 搬移「先 upsert 冷表再刪熱表」→ 中斷重跑冪等.
  - 查詢面: REST `GET /api/getArchivedTasks` / `GET /api/getArchivedMessages` + kpFunExt 同名; 後台任務頁「顯示封存」切換 (唯讀, 重試/回應停用). 封存訊息無 UI 時間軸檢視 (時間軸只呈現熱表, 稽核走 API).
  - 附件: 封存訊息之 attachments file id 仍指向 files 表/dbf 實體, getFile 照常可取.
  - **e2e 確定性**: demo 種子時間戳為固定 2026-01-01 (baseline 所需) 必逾封存齡, 故 e2e 一律以 `test/e2e-settings.json` (archiveAfterDays=0) 啟動 backend 停用掃描; 封存邏輯由 unit(archiveSweep 5 案)+api(G1) 覆蓋.

### D4. 任務 DB 不存細粒度進度, 只四態
- **決策**: state ∈ {pending, running, done, error}. 進度由 agent session 內部 (TodoWrite) 管理, 不回寫 DB. 無 updateTaskState API (見 D9).
- **理由**: 降低 agent 回寫負擔與注意力佔用 (D1 同源).
- **影響**: 前端看不到「running 30%」; 只看到「執行中」. 可接受.

### D5. respondTask 同時鏡射一筆 taskReply 訊息
- **決策**: 回應任務時 (done 或 error), 除寫 task.result/state 外, 另 post 一筆 message(kind='taskReply') 到頻道, 兩者同一 kmx 操作原子完成; task.resultMessageId 回指該訊息.
- **理由**: 頻道對話時間軸完整 (問→答都在對話流可見), 同時保留結構化任務終態.
- **影響**: 一次回應寫兩張表; 已用 kmx 序列化保證一致.

### D6. 任務生命週期三動詞共用 kmx 鎖 + state 守衛 (並發正確性)
- **決策**: claimTask / respondTask / resetTask 共用 `kmx('task:'+taskId)` 序列化. 鎖內各檢查來源 state:
  - claimTask: `state==='pending'` 否則 reject `errTaskAlreadyClaimed` (防兩 agent 搶同一任務).
  - respondTask: `state==='running'` 否則 reject `errTaskNotRunning` (防重複回應覆寫終態, 防與 resetTask lost-update).
  - resetTask: `state∈{error,running}` 否則 reject `errTaskNotResettable`.
- **理由**: 對應 CLAUDE.md 雙擊防護後端層; claimTask 有守衛則 respondTask/resetTask 必須對稱否則仍有覆寫窗.
- **影響**: 同一任務的並發操作只有一個成功, 其餘收對應 reject (預期行為非錯誤).

### D7. 訊息去重靠伺服器端游標 (channelMembers.lastSeenMessageId), agent 輪詢無狀態
- **決策**: 每個 (channel, member) 在 channelMembers 列存 `lastSeenMessageId` 游標. agent 每輪 /loop 以 getRecentMessages(afterId=游標) 取新訊息, 處理後 ackChannel 推進游標. agent 不在 session 內維護「已讀集合」.
- **理由**: 使用者以 /loop 驅動 (每輪≈重跑 prompt + session 會重啟), session 內記憶體去重必失 → 漏回新訊息或重複回. 任務面已有 state=pending 持久標記, 訊息面需對等的持久游標才對稱.
- **影響**: 多一個游標欄位 + ackChannel API; saveChannel 設 agentId 時確保 agent member 列, ackChannel 對缺列者 upsert. 訊息查詢 afterId 用 UUIDv7 + getRange 起點高效取「某 id 之後」.
- **游標 key 讀寫對稱 (關鍵)**: 游標一律以「呼叫者 token 身分 userId」為 member key 讀 (getChannelsList) 與寫 (ackChannel), **不以 agentId 為 key**. agentId 僅用於「過濾哪些頻道由此 agent 主責」. 如此即使部署時 channel.agentId 設為友善名稱 (≠ SSO 解析之 user.id), 讀(userId)寫(userId)仍命中同一列, 不會出現「讀寫不同列 → 游標永不前進 → 訊息每輪重讀」. agent 端因此不需知道自己的 user.id — 只要 ME(agentId) 等於主責頻道 channel.agentId 即可過濾出頻道, 游標由 server 依 token 自動對應.
- **自產訊息去重**: agent 同輪以 respondTask/postMessage 產生的訊息 (taskReply / 自己的 text) id 必大於本輪游標, 下輪 getRecentMessages 會讀到; agent 回話前須濾掉 `senderId===自己 && kind!=='text'` (只對他人的 text 回話), 且 ackChannel 的 lastMessageId 取「本輪讀到最新 id 與自己 post 最新 id 之較大者」, 使自產訊息納入游標. 詳 設計_agent協定 §6.

### D8. 身分權威一律取自 token, 不信任呼叫端自帶識別碼
- **決策**: 寫入的 senderId / assigneeId / 稽核 userId 一律 = getUserByToken(token) 解析之 user.id. claimTask 不收 agentId 參數 (assignee 由 token 定). senderType 由呼叫端宣告但僅供 UI 呈現, 非授權依據. getChannels 的 agentId 為**查詢過濾**參數 (過濾 channel.agentId), 非身分宣告.
- **理由**: 防冒名認領/冒名發話; 對齊「senderId=agent 自身識別碼」自證原則; 與全表 userId 稽核欄位慣例一致.
- **影響**: 單一 agent 的 assigneeId 與 taskReply senderId 必一致 (皆 token 身分), 稽核可靠.

### D9. 移除 updateTaskState, respondTask 為唯一終態入口
- **決策**: 不提供 updateTaskState. 終態 (done/error) 一律走 respondTask (一律鏡射 taskReply); error→pending 走 resetTask.
- **理由**: updateTaskState 與 respondTask(error) / resetTask 語意重疊, 且經 updateTaskState 進 error 不鏡射訊息會破壞 D5「問→答都在對話流可見」; 收斂職責更簡潔 (§2.3).
- **影響**: error 錯誤說明一律寫 task.result (人類可讀), errorKey 為選填 i18n key (供統計, 不放自由文字); 前端讀錯誤只看 result.

### D10. 內容渲染與檔案服務之 XSS 防護 (兩個安全不變量)
- **背景**: message.content / task.payload / task.result 允許 markdown (D 設計), 後端刻意以 woItems 直存不過 clearXSS (保留 markdown 原樣, procCore.mjs:19); 前端 ChatView / LayoutContentTasks 以 showdown `makeHtml` → `v-html` 渲染. showdown **不消毒** HTML, 故原始 `<script>` / `<img onerror>` / `javascript:` 連結會穿透 → 儲存型 XSS (任一具發訊權限之 human/agent 可在所有檢視者瀏覽器執行 JS, 竊 userToken).
- **決策 (不變量 1 — 渲染消毒)**: 所有 `renderMd` 一律 `DOMPurify.sanitize(mdConverter.makeHtml(c))` 後才交 `v-html` (ChatView / LayoutContentTasks 兩處). **移除此 sanitize 即 XSS 復活**; E2E-007 為其回歸守衛 (發 `<img onerror>` 訊息, 斷言 onerror 未執行 + DOM 無 onerror 屬性 + 文字保留).
- **決策 (不變量 2 — 檔案服務白名單)**: getFile (WWebTask.mjs) 僅點陣圖白名單類型 (png/jpeg/gif/webp/bmp/avif) 以原 Content-Type + `inline` 呈現; 其餘 (含 `text/html` / `image/svg+xml` 等可挾帶 script 者) 一律 `application/octet-stream` + `Content-Disposition: attachment` + `X-Content-Type-Options: nosniff` → 強制下載不於本站原點 inline 渲染. **理由**: 上傳之 type/name 不可信 (使用者自帶), 若以原 type inline 服務惡意 HTML/SVG 即為同源 XSS. api-files 測試「getFile 非白名單類型 → 強制 attachment + octet-stream + nosniff」為其守衛.
- **非問題 (已查核)**: readFile 之 `path.join(dbfDir, id)` **無路徑穿越風險** — id 須先命中 files 表 (`select({id})`) 才到 path.join, 而 file id 為 `genIDSeq()` 伺服器產生之 UUID (非使用者可控), `../` 之類 id 不存在於 files 表即被擋. files 表查詢即白名單.

### D13. RWD 採 master-detail 堆疊, 斷點 768px (2026-07-06 實作)
- **背景**: v2 UI 重設計 (§10) 將 v1 之響應式 `WDrawer` 換成固定 flex 版面 (rail 64px + 頻道樹 282px + 內容區), 窄屏時 rail + 樹即占滿寬度, 內容區被擠出畫面右緣 → RWD 失效 (390px 實測確認).
- **決策**: 以 `$store.state.isNarrow` (由 `Layout.vue` 之 `resize` 監聽維護, 斷點 `window.innerWidth < 768`) 驅動版面切換:
  - **頻道區**: 窄屏改 master-detail —— 未選頻道→頻道樹全寬; 選了頻道→聊天區全寬 + 標題列左側「返回」鈕 (`onClickBack` 清空 currentChannelId). 標題列同時以 media query 隱藏次要項 (breadcrumb / 任務 pills / agent chip) 避免溢出.
  - **後台**: 窄屏子選單改頂部橫向 tab (Channels/Members/Tasks), 資料表於下方全寬.
  - **統計 / 成員視窗**: 原生已響應 (`flex-wrap` / `WDialog` 之 `autoFullscreenWhenNarrow`), 不另處理.
- **理由**: master-detail 為聊天類應用之行動端標準模式 (先看清單→進對話→返回); 相較「縮小樹寬」可保留內容區完整可用寬度.
- **關鍵影響 (勿破壞)**: **斷點以下之版面切換一律經 `isNarrow`, 桌面 (≥768px) 版面必須與改動前逐像素相同** —— e2e 於 `viewport 1440×900` 執行 (`test/e2e-setup.mjs:208`), `isNarrow=false`, 故所有既有 pixel baseline 不受影響. 日後調整 RWD 時, 若動到 ≥768px 之版面即會觸發 baseline mismatch, 屬預期.

### D14. 成員類型在地化採 cell-render「顯示層映射」, 底層仍存原值
- **背景**: 成員資料表之 `memberType` 欄原以原始 enum (`agent` / `human`) 顯示. 其在地化不可經「轉換 `opt.rows` 資料」達成 —— `doSaveMembers` 直接讀 `opt.rows` 送 `saveChannelMember` (`LayoutContentMembers.vue`), 轉換底層值會把在地化字串寫進 DB (資料污染).
- **決策**: 以 WAggridVue 之 `cell-render` slot 僅轉**顯示**文字 (`$s.getMemberTypeText`), 底層 `row.memberType` 維持原值 `agent`/`human` 供儲存. 此與 `LayoutContentTasks` 之 `state` 欄以 `cell-render` 上色 + 在地化為**同一慣例**.
- **取捨 (明知且接受)**: ag-grid 之欄位文字過濾器比對的是**底層原值**而非渲染後文字, 故於成員表輸入「人類」過濾不會命中 (需輸入 `human`). 成員表為每頻道數列之小表, 實務上不會用打字過濾 → **後果不具體, 不修**. 若日後出現大量成員之頻道再評估.

## 已知限制 / 待討論 (合約外, 預設不做)

### L1. 頻道成員權限「弱enforcement」
- channelMembers 表記錄成員與角色, 但 v1 **不強制**「只有成員能在頻道發訊」(任何通過 token 認證者皆可發). 權限細化屬合約外.
- 若日後要強制, 在 postMessage 內加 membership 檢查即可 (單點).

### L2. 不實作本端 rate limit
- 速率限制屬 SSO 端或反向代理層責任, 本套件不重做 (對齊 w-web-api).

### L3. 單一頻道一個主責 agent
- channel.agentId 為單一主責 agent. 多 agent 共管屬合約外; 真要多 agent, 仍可各自輪詢同頻道 (claimTask 原子性保證不重複), getChannels 的 agentId 過濾以單一為準.

### L4. 任務無自動逾時/重派 (running→pending 僅人工)
- running 任務若 agent 中途死亡不會自動回 pending (自動恢復屬合約外, 未要求). 人類可手動 resetTask (running→pending) 救援停滯任務 — 此為 resetTask 的選用來源狀態 (error→pending 才是標準重試, 屬合約內). 後續若需要可加 timeClaim 逾時掃描.
- **確認 (2026-07-01)**: 任務卡在 running 為單一任務列之孤立問題, 不影響頻道其餘運作 — postMessage 建新訊息、開新任務 (新 pending 任務列) 皆與舊任務狀態無關; agent 只要仍存活並輪詢 getRecentMessages/getRecentTasks, 新訊息與新任務照常送達與可 claimTask. 僅該筆卡住的任務本身需人工 resetTask 才能重新被 claim (claimTask 守衛 state==='pending', running 中不可再搶). 維持現狀 (不加自動逾時掃描).

### L5. channelMembers 管理 UI (v2 已補: 各頻道內建成員管理入口)
- 後端 procCore 與 kpFunExt 提供 getChannelMembers / saveChannelMember / deleteChannelMember.
- **v2 決策 (2026-07-01)**: 各頻道聊天頁 (ChatView) 標題列加入「成員」入口, 開啟 WDialog 包覆既有 LayoutContentMembers (後台管理沿用之元件), 以新增 `fixedChannelId` prop 綁定當前頻道 — 給定時隱藏頻道選擇器, 直接管理該頻道成員, 免再選頻道. 後台管理「成員」子選單 (含頻道選擇器, 可切換管理任一頻道) 仍保留, 兩入口共用同一元件與 API, 僅差 `fixedChannelId` 是否綁定.
- UI 內無額外權限判斷: 能通過 SSO 進站者一律視為管理員 (與現行「認證即信任」模型一致, 對齊 L1).
- 主責 agent 由 channel.agentId 設定 + saveChannel 自動建 agent member 列; 游標列由 ackChannel upsert.
