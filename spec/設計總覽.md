# w-web-task 設計總覽

> 任務安排調度套件. 架構沿用姊妹專案 (w-web-sso / w-web-perm / w-web-api): mjs + Vue2 + lmdb + w-serv-hapi + w-serv-orm + w-component-vue + wsemi. 認證委外 SSO (w-web-sso).
>
> 本版已納入設計對抗性審查 (review-design) 確認之修正: 訊息游標 (afterId + channelMembers 水位 + ackChannel)、resetTask 補入 API、移除 updateTaskState (respondTask 為唯一終態入口)、respondTask 加 running 守衛、getRecentTasks 加 order、移除 priority、assigneeId 取自 token、asTask 契約.

## 1. 系統定位

提供「頻道 (channel) → 對話訊息 (message) → 任務 (task)」三層模型, 讓人類成員在頻道內對話, 並可把對話「擴充指向任務」. AI agent 以主責頻道為單位輪詢, 收到新訊息/任務後執行並回應. 任務「有收與回, 回完即結束」.

- **人類**: 透過前端 Web UI (Vue2) 管理頻道、發訊、開任務、檢視任務回應.
- **AI agent**: 透過對外 HTTP REST API 輪詢頻道、認領任務、回應任務 (CLI session 內以 curl/fetch 即可呼叫, 無需長連線).
- **認證**: 一律委外 w-web-sso (token 由 SSO 核發). 本套件不存 users/tokens, 只驗 token → user. 開發捷徑 `token=sys` (NODE_ENV !== production).

## 2. 架構分層 (沿用姊妹專案)

```
srv.mjs (薄進入點)
  → server/WWebTask.mjs (主類別, 組裝一切; 簽名仿 WWebApi: WWebTask(WOrm,url,db,getUserByToken,verifyClientUser,verifyAppUser,opt))
      ├─ 讀 settings.json (JSON5, 由 srv.mjs JSON5.parse 後併入 opt)
      ├─ src/schema/index.mjs → WServOrm(ds, WOrm, url, db) → { woItems, procOrm }
      ├─ srLog.mjs           w-syslog 包裝 (.info/.warn/.error) + maskTok
      ├─ procLang.mjs        i18n 字典 { eng:{...}, cht:{...} }
      ├─ procCore.mjs        核心業務 (channel/member/message/task CRUD + 任務生命週期)
      │     內含 kmx = pmKeyMutex() 作 critical 路徑序列化
      └─ WServHapiServer({
            port, pathStaticFiles, apiName:'api',
            apis: [ ...HTTP REST (對外 agent + w-ui-loginout) ],
            verifyConn,            // 連線層: token 認證 (p.checkToken)
            getUserIdByToken,      // 回 token 解析之 userId
            useDbOrm:true, kpOrm:woItems, operOrm:procOrm, tableNamesExec, methodsExec, tableNamesSync,
            kpFunExt: { ...WebSocket RPC (對內前端) },
            fnTableTags: 'tableTags-web-task.json',
        })
```

- **對外 agent 面 = HTTP REST (`apis` 陣列)**: 適合輪詢 (polling), 無持久連線, CLI 易呼叫. token 由 `?token=` query (GET) 或 payload (POST) 帶入, 各 handler 以 `getAndVerifyAppUser(token)` 守門, 用 `pm2resolve(core)()` 包裝回傳.
- **對內前端面 = kpFunExt (WebSocket RPC) + useDbOrm 直通**: 前端 `$fapi.xxx()` 即時呼叫. kpFunExt 第一參 `_t` 為 userId, 業務參數從第二參起; 成功直接 return, 錯誤 `reject('errKey')`, 統一 wrapErrLog 記 log. **同步策略**: 純 CRUD 表 (channels/channelMembers) 走 ORM change 事件即時推送 (tableNamesSync=tableNamesExec); 而 messages/tasks **不走 ORM 同步**, 前端一律以 `$fapi.getMessages/getTasks` 按需拉取 (選頻道/發送後重抓; 另有 ChatView 開窗期間每 5 秒之靜默輪詢, 使 agent 回應無須切頻道即可見), 存本地渲染 — 因 messages/tasks 不在 tableNamesExec (其寫入須走 kpFunExt 業務函式保 kmx/守衛), 放進 sync 會使同步引擎找不到 select binding 而報錯.
- 回傳契約: 查詢成功回數據, 操作成功回 i18n key 或結果物件; 錯誤一律 `reject('errKey')` 字串 (procLang 字典 key), 不回明文. 後端同步 `srLog.error({ event, err })`.

## 3. 資料表 (7 張)

> 型別: STRING / INTEGER / TEXT. 每張表沿用稽核欄位 `userId / timeCreate / userIdUpdate / timeUpdate / isActive('y'|'n')`. 主鍵 id 用 `genIDSeq()` (UUIDv7, 時序遞增 → 字典序即時序; LMDB getRange / select 全表即插入序). 時間用 `nowms2str()`.

### 3.1 channels — 頻道

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | STRING(pk) | 主鍵 |
| order | INTEGER | 排序 |
| name | STRING | 頻道名稱 |
| description | TEXT | 頻道描述 |
| agentId | STRING | 主責 agent 識別碼 (輪詢此頻道之 agent; 空字串=未指派) |
| ownerId | STRING | 建立者 (SSO user id) |
| userId / timeCreate / userIdUpdate / timeUpdate / isActive | — | 稽核欄位 |

### 3.2 channelMembers — 頻道成員 (兼 agent 已讀水位)

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | STRING(pk) | 主鍵 |
| order | INTEGER | 排序 |
| channelId | STRING | → channels.id |
| memberId | STRING | 成員識別碼 (SSO user id 或 agent id) |
| memberType | STRING | `human` / `agent` |
| role | STRING | `owner` / `member` |
| lastSeenMessageId | STRING | 該成員「已讀/已處理」的最後一則訊息 id (游標水位). agent 輪詢用此跨 session 續接, 由 ackChannel 推進 |
| userId / timeCreate / userIdUpdate / timeUpdate / isActive | — | 稽核欄位 |

> saveChannel 設定 channel.agentId 時, 後端確保該 agent 有對應 channelMembers 列 (memberType='agent'); ackChannel 亦對缺列者 upsert, 使游標有落點.

### 3.3 messages — 對話訊息

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | STRING(pk) | 主鍵 (時序遞增 → 即發話順序; 游標比較基準) |
| channelId | STRING | → channels.id |
| senderId | STRING | 發送者識別碼 (token 解析之 user/agent id, 權威來源) |
| senderType | STRING | `human` / `agent` (呼叫端宣告, 僅供 UI 呈現, 非授權依據) |
| kind | STRING | `text`(一般對話) / `task`(開任務) / `taskReply`(任務回應) |
| content | TEXT | 訊息內容 (允許 markdown) |
| taskId | STRING | kind=`task`→spawned task.id; kind=`taskReply`→被回應 task.id; 否則 '' |
| userId / timeCreate / userIdUpdate / timeUpdate / isActive | — | 稽核欄位 |

### 3.4 tasks — 任務

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | STRING(pk) | 主鍵 |
| channelId | STRING | → channels.id |
| messageId | STRING | → messages.id (起源訊息, 即「收」) |
| title | STRING | 任務標題 |
| payload | TEXT | 任務需求內容 (「收」, 允許 markdown) |
| state | STRING | `pending` / `running` / `done` / `error` |
| assigneeId | STRING | 認領之 agent id (claimedBy); **權威來源 = claimTask 時 token 解析之身分**, 非呼叫端自由帶入 |
| result | TEXT | 任務回應內容 (「回」, 允許 markdown; **error 時亦寫此欄為人類可讀錯誤說明**) |
| errorKey | STRING | state=error 時**選填**之 i18n key (供統計/log; 自由文字說明一律走 result, 不混入此欄) |
| resultMessageId | STRING | → messages.id (回應所產生之 taskReply 訊息) |
| timeClaim | STRING | 認領時間 |
| timeDone | STRING | 結束時間 (done/error 寫入) |
| userId / timeCreate / userIdUpdate / timeUpdate / isActive | — | 稽核欄位 |

> 不設 priority 欄位 (v1 採純 FIFO, 取最舊 pending; LMDB 後加欄位無 migration 成本, 真有多優先序需求再加).

### 3.5 files — 圖台檔案 (附件實體 metadata)

> 附件不 base64 內嵌訊息 (避免對話過大 / 耗 agent token), 改上傳存圖台取 id, 訊息只存 id (見 §10.3). 實體 bytes 存 `./dbf/<id>` (無副檔名).

| 欄位 | 型別 | 說明 |
|---|---|---|
| id | STRING(pk) | 主鍵 |
| name | STRING | 原始檔名 |
| type | STRING | MIME 類型 |
| size | INTEGER | 檔案大小 (bytes) |
| ext | STRING | 原副檔名 (選填) |
| userId / timeCreate / userIdUpdate / timeUpdate / isActive | — | 稽核欄位 |

### 3.6 messagesArchive — 訊息封存冷表

> messages 之封存冷表, schema 與 `funNew` 與 3.3 messages **完全一致** (schema spread), 僅表名不同. 過齡舊訊息由 `procCore.archiveSweep` 搬入, 熱表僅留活躍資料 (封存機制見 `設計要點與取捨.md` D12). 欄位同 3.3 (id / channelId / senderId / senderType / kind / content / taskId + 稽核欄位).

### 3.7 tasksArchive — 任務封存冷表

> tasks 之封存冷表, schema 與 `funNew` 與 3.4 tasks **完全一致** (schema spread), 僅表名不同. 終態 (done/error) 且過齡之任務由 `procCore.archiveSweep` 搬入, 熱表僅留活躍資料 (封存機制見 `設計要點與取捨.md` D12). 欄位同 3.4 (id / channelId / messageId / title / payload / state / assigneeId / result / errorKey / resultMessageId / timeClaim / timeDone + 稽核欄位).

## 4. 關聯

```
channels ──1:N── messages        (messages.channelId)
channels ──1:N── tasks           (tasks.channelId)
channels ──1:N── channelMembers  (channelMembers.channelId)
messages ──1:0..1── tasks        (message kind='task' → tasks.messageId 指回起源訊息)
tasks    ──回應──→ messages       (tasks.resultMessageId 指向 taskReply 訊息)
channelMembers.lastSeenMessageId → messages.id  (該成員游標水位)
```

- LMDB 無外鍵約束, 關聯以字串欄位 + application-layer 查詢表達 (兩次 select).
- 「對話可擴充指向任務」: postMessage(asTask=true) → 同一 kmx 內原子寫一筆 message(kind='task') + 一筆 task(state='pending', messageId=該訊息).
- 「回完即結束」: respondTask → 寫 task.result + state(done/error) + timeDone, 並 post 一筆 message(kind='taskReply', taskId=該任務) 使對話時間軸完整; task.resultMessageId 指向它.

## 5. 任務狀態機

```
                claimTask                   respondTask(done)
  [pending] ─────────────────▶ [running] ────────────────────▶ [done]   (終態)
     ▲                            │
     │                            └── respondTask(error) ────▶ [error]  (終態)
     │                                                            │
     ├──────────── resetTask (error→pending, 標準重試) ───────────┘
     └──────────── resetTask (running→pending, 人工救援停滯任務; L4 選用) ──
```

- `pending`: 任務建立, 待 agent 認領 (對應使用者口語的 "none").
- `running`: agent 已認領 (claimTask: kmx 鎖內檢查 state==='pending' 才 pending→running, 蓋 assigneeId/timeClaim).
- `done`: respondTask 寫 result + state='done'. 終態.
- `error`: respondTask 寫 result(錯誤說明) + state='error' + 選填 errorKey. 終態.
- `resetTask`: error→pending (標準重試) 或 running→pending (人工救援停滯任務). kmx 鎖內檢查來源 state∈{error,running}, 原子清空殘留欄位 (result/errorKey/assigneeId/timeClaim/timeDone/resultMessageId **以空字串 ''** 清, 因底層 lodash merge 不清 undefined) 後設 pending.
- **並發守衛 (對稱 D6)**: claimTask 鎖內 `state==='pending'` 否則 reject `errTaskAlreadyClaimed`; respondTask 鎖內 `state==='running'` 否則 reject `errTaskNotRunning` (防重複回應覆寫終態 / 與 resetTask lost-update); resetTask 鎖內 `state∈{error,running}` 否則 reject `errTaskNotResettable`. 三者共用 `kmx('task:'+taskId)` 同一鎖序列化.

## 6. API 目錄

### 6.1 對外 agent — HTTP REST (`apis` 陣列)

> token 由 GET `?token=` 或 POST payload 帶入; handler 內 `getAndVerifyAppUser(token)` 驗身分. 回 `{ state, msg }` (pm2resolve). 查詢成功回數據, 操作成功回 key 或結果, 錯誤回 errKey. assigneeId/senderId 一律取自 token 解析身分.

| method | path | 功能 | 主要參數 |
|---|---|---|---|
| GET | /api/getChannels | 查頻道清單 (可依主責 agent 過濾; 為 agent 過濾時, 每筆附該 agent 之 lastSeenMessageId 游標) | token, agentId? |
| GET | /api/getRecentMessages | 查頻道訊息: 給 afterId 則回該 id 之後**全部新訊息**(時序, 上限 500 筆); 未給則回最近 n 筆 (供初次/人類檢視) | token, channelId, n=20, afterId? |
| GET | /api/getRecentTasks | 查頻道任務: state 過濾 (如 pending); order='asc' 取最舊優先 (agent FIFO 消化), 'desc' 取最近優先 (人類檢視, 預設) | token, channelId, n=20, state?, order='desc' |
| GET | /api/getArchivedTasks | 查頻道封存任務 (冷表 tasksArchive, 稽核/回顧用; 見 D12) | token, channelId, n=20, order='desc' |
| GET | /api/getArchivedMessages | 查頻道封存訊息 (冷表 messagesArchive, 稽核用; 見 D12) | token, channelId, n=20 |
| GET | /api/getTask | 查單一任務 | token, taskId |
| POST | /api/postMessage | 傳訊至頻道 (asTask=true 同時開任務) | token, channelId, content, asTask?, title?, senderType? |
| POST | /api/claimTask | 認領任務 (pending→running; assigneeId=token 身分) | token, taskId |
| POST | /api/respondTask | 回應任務 (唯一終態入口; running→done/error; 寫 result + 發 taskReply) | token, taskId, result, state='done', errorKey? |
| POST | /api/resetTask | 重置任務 (error/running→pending; 清殘留欄位) | token, taskId |
| POST | /api/ackChannel | 推進呼叫者 (agent) 在該頻道之已讀游標 lastSeenMessageId | token, channelId, lastMessageId |
| GET | /api/getUserByToken | (w-ui-loginout 用) token→user | token |

### 6.2 對內前端 — kpFunExt (WebSocket RPC)

> 前端 `$fapi.xxx(token, ...)` 呼叫. 業務函式與對外面**共用同一 procCore**, 僅傳輸層與認證不同. 成功直接 return, 錯誤 `reject('errKey')`, wrapErrLog 記 log.

| 函式 | 功能 |
|---|---|
| getWebInfor() | web 設定 + kpLang (免 token) |
| getChannelsList(token) | 頻道清單 (前端管理用) |
| saveChannel(token, row) / deleteChannel(token, id) | 頻道增改 / 刪 (saveChannel 設 agentId 時確保 agent member 列) |
| getChannelMembers(token, channelId) | 成員清單 |
| saveChannelMember(token, row) / deleteChannelMember(token, id) | 成員增改 / 刪 |
| getMessages(token, channelId, n, afterId) | 頻道訊息 (時間軸; afterId 同對外面語意) |
| postMessage(token, channelId, content, opt) | 發訊 (opt.asTask 開任務, opt.title, opt.senderType) |
| getTasks(token, channelId, n, state, order) | 任務清單 (state/order 同對外面) |
| getArchivedTasks(token, channelId, n, order) | 封存任務清單 (冷表 tasksArchive, 唯讀檢視; 見 D12) |
| getArchivedMessages(token, channelId, n) | 封存訊息清單 (冷表 messagesArchive, 稽核用; 見 D12) |
| getTask(token, taskId) | 單一任務詳情 |
| claimTask(token, taskId) | 認領 (前端通常不用, 供測試/手動) |
| respondTask(token, taskId, result, state, errorKey) | 回應 (人類也可代回; running 守衛) |
| resetTask(token, taskId) | 重置 (前端 error/停滯任務「重試」按鈕) |
| ackChannel(token, channelId, lastMessageId) | 推進游標 (前端通常不用) |

> 註: useDbOrm:true 使前端可直通 ORM select/insert/save/del (供 ag-grid 同步顯示), 但任務生命週期 (claim/respond/reset)、開任務 (postMessage asTask)、游標 (ackChannel) 等需原子性/多表寫入者一律走 kpFunExt 業務函式, 不走 ORM 直通.

### 6.3 postMessage 的 asTask 契約

- `asTask=true`: 原子 (kmx) 寫 message(kind='task', content) + task(state='pending', payload=content, messageId=該訊息, title). title 缺省時取 content 首行 (截斷至合理長度) 為 title. 回傳含新 task.id.
- `asTask=false` (預設): 僅寫 message(kind='text', content); 忽略 title.
- senderId 一律取自 token 解析身分; senderType 由呼叫端宣告 (預設 'human'), 僅供 UI 呈現.

## 7. 認證模型 (委外 SSO)

- `srv.mjs` 注入 `getUserByToken(token)`: `token==='sys' && NODE_ENV!=='production'` → 回管理員虛擬 user (開發捷徑); 否則 GET `${ssoBaseUrl}/api/getSsoUserInfor?token=${ssoAppToken}&key=token&value=${token}` 取 user, 失敗回 `{}`.
- `verifyConn`: 連線層僅認證 (p.checkToken: token→user 欄位驗齊), 不授權. 授權屬套件使用者 (verifyClientUser/verifyAppUser) 責任 (對齊 w-web-api ADR-022).
- HTTP handler 以 `getAndVerifyAppUser(token)` 守門 (agent/app 面); 前端登入走 w-ui-loginout → `/api/getUserByToken` → getAndVerifyClientUser.
- **身分權威**: 所有寫入的 senderId / assigneeId / 稽核 userId 一律取自 token 解析之 `user.id`, 不信任呼叫端自帶的 agentId/senderId (getChannels 的 agentId 為**查詢過濾**參數, 非身分宣告).
- 前端 localStorage key `wtask:userToken`.

- **頻道區** → **左2 頻道樹 (282px)**: 搜尋過濾框 + **階層 tree** (依 channels.levels 分組: 專案 > 群組 > #頻道; 含展開/收合、在線小圓點 (channel.agentId 非空)、active teal 底); 點頻道載入右側. (註: 「未讀數徽章」v2 未實作 — 需人類已讀追蹤 (目前 channelMembers.lastSeenMessageId 僅由 agent ackChannel 推進, 無人類進站推進機制); 屬後續可加, 視覺基準 mockup 之徽章為示意.)
  - **右側**: 頻道標題列 (# 名稱 + 麵包屑(levels) + 任務狀態 pills + agent 在線 chip) → **對話時間軸** (訊息氣泡: 人類/agent、task/taskReply 徽章卡、圖片附件縮圖) → **composer** (圖片預覽縮圖含 ×、textarea「可貼多圖, Enter 發送/Shift+Enter 換行」、附圖鈕、開成任務 toggle、發送鈕).

- **統計資訊** → 右側 dashboard: 統計卡 (頻道數 / 訊息數 / 任務各狀態數) + 視需要 per-channel 細項.

### 10.3 圖台 / 檔台 (file store) — 新增
> 動機: 附件若 base64 內嵌訊息會使對話過大、消耗 agent token. 改為「上傳前先存圖台取 file id, 訊息只存 id」.

- **新表 `files`**: `id`(pk, genIDSeq) / `name`(原始檔名 STRING) / `type`(mime STRING) / `size`(INTEGER bytes) / `ext`(原副檔名 STRING, 選填) + 稽核欄位. **實體 bytes 存 `./dbf/<id>` (無副檔名)**.
- **messages.attachments** (TEXT, JSON array of file id 字串): 訊息只存 id, 不存 base64.
- **API (對外 agent HTTP + 對內 kpFunExt 共用 procCore)**:
  - `POST /api/uploadFile` {token, name, type, dataBase64} → 寫 ./dbf/<id> + insert files → 回 `{id, name, type, size}`. (上傳)
  - `GET /api/getFile?token=&id=` → 串流檔案 bytes, 設正確 `Content-Type` (供 agent/前端**看圖/下載**; 非 pm2resolve JSON, 直接回二進位).
  - `GET /api/getFileInfo?token=&id=` → files metadata (供 agent 知道原始檔名/type/size).
  - kpFunExt: `uploadFile` / `getFileInfo` (前端); getFile 走 HTTP 二進位.
- **agent 可用性**: agent 拿到訊息的 attachments(file ids) → `getFileInfo` 知檔名/type → `getFile` 取 bytes 檢視. 文件化於 README/agent 指南.

### 10.4 channels.levels (階層欄位) — 新增
- **channels 加 `levels`** (STRING, 英文句點 `.` 分隔, 如 `電商平台改版.後端組`): 前端頻道樹依此分組分類; 麵包屑顯示亦用之. 空字串 = 未分類 (歸「未分類」群).

### 10.5 getStats — 新增
- `getStats(token)` (HTTP + kpFunExt): 回 `{ channels, messages, tasks:{pending,running,done,error,total} , (可選 perChannel) }` 供統計頁.

### 10.6 對外 agent API 增補總表 (補於 §6.1)
| method | path | 功能 |
|---|---|---|
| POST | /api/uploadFile | 上傳檔案 → 回 file id/meta |
| GET | /api/getFile | 串流檔案 bytes (看圖/下載) |
